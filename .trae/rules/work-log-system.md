# 开发工作日志规范

## 核心原则

1. **每天一个文件**，记录当天的开发工作
2. **按功能模块组织**，清晰展示工作成果
3. **内容详细、结构化**，像正式开发日志一样

## 目录结构

```
.trae/
└── memory/
    └── work-log/              # 工作日志目录
        └── {YYYY-MM}/         # 按月分组，如 2026-04/
            └── {YYYY-MM-DD}.md  # 每天一个文件，如 2026-04-26.md
```

## 文件命名规则

- 按月创建文件夹，格式：`YYYY-MM/`
- 每天一个文件，格式：`YYYY-MM-DD.md`
- 示例：`memory/work-log/2026-04/2026-04-26.md`

## 日志内容格式

```markdown
# {YYYY-MM-DD} 工作日志

## 功能模块：{模块名称}

### 完成内容
1. {具体完成的工作1}
2. {具体完成的工作2}
3. {具体完成的工作3}

### 技术要点
- {技术点1}
- {技术点2}
- {技术点3}

### 重要经验
- {经验总结1}
- {经验总结2}

### 后续优化
- {优化点1}
- {优化点2}
```

## 字段说明

| 字段 | 说明 | 必填 |
|------|------|------|
| 日期标题 | `# YYYY-MM-DD 工作日志` | 是 |
| 功能模块 | 当前开发的功能/模块名称 | 是 |
| 完成内容 | 当天完成的具体工作，用序号列出 | 是 |
| 技术要点 | 用到的关键技术、API、方案 | 是 |
| 重要经验 | 踩坑记录、解决方案、心得 | 否 |
| 后续优化 | 待改进点、优化思路 | 否 |

## 示例

```markdown
# 2026-04-26 工作日志

## 功能模块：用户登录模块

### 完成内容
1. 完成登录页面 UI 开发
2. 实现手机号验证码登录功能
3. 集成微信一键登录
4. 编写登录状态管理（Pinia）

### 技术要点
- 使用 uni.login() 获取微信 code
- 验证码倒计时使用 useCountdown hook
- 登录 token 存储在 localStorage

### 重要经验
- 微信小程序获取手机号需要企业认证
- 验证码发送接口需要防刷机制

### 后续优化
- 添加登录失败重试机制
- 优化登录加载动画
```

## AI 执行规则

### 写入触发条件

| 场景 | 动作 |
|------|------|
| 完成功能开发 | 记录到当天日志的"完成内容" |
| 修复 Bug | 记录问题原因和解决方案到"重要经验" |
| 使用新技术/方案 | 记录到"技术要点" |
| 发现可优化点 | 记录到"后续优化" |

### 写入规则

1. **自动创建文件**：如果当天日志文件不存在，自动创建
2. **自动创建目录**：如果当月目录不存在，自动创建
3. **追加模式**：同一天多次对话，追加到同一个文件
4. **多模块支持**：一天内多个功能模块，分别用 `## 功能模块` 分隔

### 内容规范

- **完成内容**：用序号列表，描述具体做了什么
- **技术要点**：用无序列表，记录关键技术和实现细节
- **重要经验**：记录踩坑、解决方案、注意事项
- **后续优化**：记录 TODO、优化思路、待办事项

## 特殊情况处理

### 跨天开发同一个功能

每个文件独立记录，但可以在"完成内容"中注明进度：
```markdown
### 完成内容
1. 用户登录模块 - 完成 80%（验证码接口联调中）
```

### 修复 Bug

在"完成内容"记录修复，在"重要经验"记录原因：
```markdown
### 完成内容
1. 修复登录页面白屏问题

### 重要经验
- 白屏原因：onLoad 生命周期中调用未初始化的数据
- 解决：使用 onReady 或在 onLoad 中添加空值判断
```

### 无实质产出

如果当天只是讨论、调研，没有代码产出：
```markdown
# 2026-04-26 工作日志

## 功能模块：技术调研

### 完成内容
1. 调研微信支付接入方案
2. 对比 uni-pay 和原生接入的优缺点

### 技术要点
- uni-pay 支持多平台统一接入
- 原生接入需要分别处理微信小程序和 APP

### 后续优化
- 待确定最终方案后实施
```
