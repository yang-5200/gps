# 开发工作日志规范

## 1. 核心原则

1.  **每日记录**：每天一个独立文件，真实记录当天的开发轨迹。
2.  **模块化组织**：按功能模块划分区块，清晰展示每个功能的进度。
3.  **结构化内容**：严格遵循 Markdown 格式，确保内容易于 AI 读取和检索。
4.  **只增不改**：保持日志的原始时间线，修正错误时使用删除线。
5.  **代码关联**：推荐使用**相对路径**链接记录代码变更（如 `[basename](../path/to/file)`），以确保项目移动或在不同设备打开时链接依然有效。

---

## 2. 目录与命名规范

### 2.1 目录结构
```text
.trae/
└── memory/
    ├── YYYY-MM-DD.md       # 每日日志
    └── YYYY-MM-summary.md  # 月度总结
```

### 2.2 命名规则
| 类型 | 格式 | 示例 |
| :--- | :--- | :--- |
| **每日日志** | `YYYY-MM-DD.md` | `2026-04-30.md` |
| **月度总结** | `YYYY-MM-summary.md` | `2026-04-summary.md` |

## 3. 日志内容定义

### 3.1 基础模板
```markdown
# YYYY-MM-DD 工作日志

## 功能模块：{模块名称}

> 类型：`新功能` | `Bug修复` | `重构` | `优化` | `调研` | `协调`
> 状态：`进行中` | `已完成` | `已阻塞`

### 完成内容
1. {具体工作项 1}
2. {具体工作项 2}

### 技术要点
- {关键技术/API/方案，需包含代码引用链接}

### 重要经验
- {避坑指南/解决方案/心得总结}

### 阻塞项 & 优化（如有）
- [P0] {阻塞描述} → 影响范围
- [P2] {待优化点}

### 关联记录
- 前置：[YYYY-MM-DD](./YYYY-MM-DD.md) - {简述}
```

### 3.2 字段说明
| 字段 | 必填 | 说明 |
| :--- | :--- | :--- |
| **功能模块** | 是 | 当前开发的功能/子系统名称 |
| **类型/状态** | 是 | 标记工作属性及当前进展阶段 |
| **完成内容** | 是 | 详细记录当天的产出，建议使用序号 |
| **技术要点** | 是 | 记录核心实现逻辑及关联的代码文件链接 |
| **优先级定义** | - | P0: 紧急阻塞 | P1: 影响后续 | P2: 建议优化 | P3: 备选 |

---

## 4. AI 执行逻辑

### 4.1 写入规则
| 触发场景 | 执行动作 |
| :--- | :--- |
| **会话启动** | **主动读取最近 2-3 天的日志文件以恢复上下文状态** |
| **功能开发/修复** | 实时记录「完成内容」与「技术要点」 |
| **遇到问题** | 将原因及解法记录至「重要经验」 |
| **发现隐患** | 记录至「后续优化」并标注优先级 |
| **月底最后一天** | **主动提醒并生成本月 `YYYY-MM-summary.md`** |

### 4.2 文件操作
1.  **环境检查**：在 Chat 模式下，AI 应主动检查 `.trae/memory/` 目录状态。
2.  **上下文恢复**：如果检测到多个工具在操作，优先以日志文件内容为准。
3.  **内容生成**：AI 以代码块形式生成符合规范的日志内容，并引导用户应用更改。
4.  **会话汇总**：在复杂任务结束后，进行一次性汇总写入，避免记录碎片化。

## 5. 多编辑器协作规范（跨工具同步）

### 5.1 共享内存机制
当同时使用 Trae, Cursor, CodeBuddy 等多个 AI 编辑器时，**`.trae/memory/` 目录即为它们的“共享大脑”**。

### 5.2 协作流程
1. **统一规约**：将此 `work-log-system.md` 规范告知所有 AI 工具，确保它们都向该目录写入日志。
2. **强制同步**：
   - **开始前**：要求 AI 先执行 `Read` 操作读取最新的日志。
   - **结束后**：要求 AI 必须更新日志，记录下它刚才做了什么。
3. **版本控制**：建议将 `.trae/memory/` 纳入 Git 管理。这样在不同设备间切换时，只需 `git pull` 即可同步所有 AI 的工作轨迹。
